**Status:** Approved by user (pending spec review)

## Vision

A persistent online survival world — like Minecraft, GTA, RDR2, and Rust — but underpinned by real physics, chemistry, and biology. The world is not hand-crafted by designers. It operates by real rules and was generated by real simulation. Players join a world that already exists and has history. They spawn with nothing and survive by understanding and working with the actual rules of the world.

The reference is the "Overgeared" / SAO concept: a game world so realistic it feels like a real alternate universe. The simulation IS the world — not a feature players access through a menu.

---

## Core Principles

1. **The laws are always real. Only the resolution scales.** No region of the world uses fake physics. Distant regions compute at lower resolution but follow the same rules.
2. **Discovery through doing, not menus.** Players learn that bronze is stronger than copper by experimenting with fire and metal — not by clicking a tech tree node.
3. **No dead ends.** Every implemented feature must be fully wired end-to-end. If you can open the inventory, you must be able to put something in it. If you can craft a tool, you must be able to equip it and use it.
4. **Full vertical slices.** Each feature is built complete before moving to the next. Gather → inventory → craft → equip → use. One flow at a time, working all the way through.

## Section 1: The World

### What it is
A persistent sphere planet on a 24/7 server. The world exists before players arrive and continues after they log off. It was generated by real rules, not placed by a designer.

### World generation (procedural, precomputed, seeded)
The world is generated once from a deterministic seed before any player joins:
- **Geology**: Simplified tectonic plate simulation → mountain ranges, ocean basins, river paths. Ore deposits placed by real geological rules (copper near volcanic zones, iron in sedimentary layers, clay along riverbanks, coal in ancient organic layers).
- **Biomes**: Temperature and moisture maps produce biome boundaries. Cold wet = boreal forest. Hot dry = desert. No manual placement.
- **Civilization history**: NPC settlement simulation run at extreme time scale before world opens. Settlements have territory, histories, trade relationships baked in.
- Same seed always produces the same world. Server operators choose the seed.

### What makes it feel real
- Rivers carved by erosion from geological simulation
- Resources are where chemistry and geology put them — not evenly distributed
- World has scars: burned forests grew back as meadows, depleted mines have empty tunnels, player ruins remain
- Existing NPC civilizations with history, grudges, trade routes, territory

### Starting condition
Players choose spawn zone:
- **Wilderness spawn** — deep nature, maximum freedom, maximum danger, no nearby help
- **Civilization spawn** — near an existing NPC settlement, easier access to shelter and trade, but you are an outsider who must earn trust

Either way: you spawn with nothing. Stone age start regardless of what era the world is currently in.

**Stone age start defined mechanically:**
- Empty inventory
- No tools
- Bare hands can gather loose surface rocks and fallen sticks
- Knowledge of basic fire-making (flint + dry tinder) is innate — players don't need to "discover" fire, but they do need to gather the materials
- Cannot use metal tools until they have personally crafted them (no buying finished weapons at start)

### Session persistence (auth + state)
- Clerk handles authentication (existing system)
- Player state saved to Neon DB every 60 seconds and on clean logout
- State stored: position, inventory contents, vitals values, skill levels, bedroll location, criminal record
- On reconnect: player spawns at last saved position with saved inventory intact
- Death wipes inventory but preserves position data (respawn at bedroll location)

### What this replaces
- The god-game observer framing is removed entirely
- No "Evolution Points" spending menu
- No "Tech Tree" as a god-mode unlock screen
- No bootstrap timelapse cinema (replace with simple "World is initializing — please wait" on fresh server)

## Section 2: The Simulation Engine

### Adaptive real-world simulation (Option 1+3 combined)

Every part of the world is divided into cells. Each cell stores: temperature, pressure, chemical composition (per-element concentrations), moisture, light level. The physical laws are real and apply identically everywhere. Resolution scales with distance from the nearest player:

| Tier | Distance from player | Cell size | Tick rate | What is simulated |
|------|---------------------|-----------|-----------|-------------------|
| Local | 0–50m | 0.5m | 20/sec | Full: fire O₂, heat conduction, bacterial growth, reactions |
| Nearby | 50–500m | 5m | 2/sec | Fire spread, river flow, temperature gradients |
| Regional | 500m–5km | 50m | 0.2/sec (1 per 5s) | Weather, major temperature fronts, NPC movement |
| Distant | 5km+ | 500m | 0.02/sec (1 per 50s) | Civilization behavior, large-scale climate |

**Cell promotion/demotion:** When a player moves into a tier boundary, cells subdivide (promoting to higher resolution). When no player has been within range for 30 seconds, cells merge back (demoting). Values are averaged during merge, interpolated during split.

### Chemistry
There is no recipe list. Crafting works because chemistry works:
- Reactions are evaluated using the Arrhenius equation: `k = A × e^(-Ea/RT)` where T is cell temperature, Ea is the activation energy for the reaction
- When two or more reagents are co-located in a cell at sufficient temperature, the reaction proceeds at rate k
- Example: copper ore (Cu₂S) + charcoal (C) + heat → copper (Cu) + SO₂ (gas). This is the real reduction reaction.
- Example: copper (Cu) + tin (Sn) at high temperature → bronze alloy. The eutectic point (88% Cu / 12% Sn) is modeled.
- Players who study real chemistry can discover combinations the designers never scripted. Players who don't can still figure it out by experimentation.

**How a player initiates a reaction (crafting interaction model):**
Everything happens in the physical world — no crafting UI table:
- **Fire**: Gather dry tinder + wood, place in world, strike flint nearby → sim grid checks O₂ and temperature conditions, ignites if conditions allow
- **Smelting**: Build a furnace structure (enclosed stone space), place ore + charcoal inside, light fire → temperature rises, chemistry reactions run automatically when threshold reached, refined metal appears
- **Tool crafting**: Hold a rock in hand, strike against another rock on the ground → chipping reaction produces flint flakes. Combine flint + wood + fiber by holding and performing the bind action → stone tool
- Players physically interact with materials in the world. The simulation evaluates the physics/chemistry of those interactions and produces results.

### Thermodynamics
- Temperature tracked per cell using Fourier heat conduction: `dT/dt = k × ∇²T`
- Materials have real specific heat capacity (rock = 840 J/kg·K, iron = 449 J/kg·K, water = 4186 J/kg·K)
- Phase changes at real temperatures: water freezes at 0°C, boils at 100°C (adjusted by altitude using barometric pressure formula)
- Heat radiates from hot surfaces using Stefan-Boltzmann: `P = εσT⁴`

### Biology / Pathogens
- Food spoils via logistic bacterial growth model: `dN/dt = rN(1 - N/K)` where N is bacteria count, r is growth rate (temperature-dependent), K is carrying capacity
- Cooked food lasts longer because heat reduces initial N to near zero
- Wounds: tissue damage opens infection risk. Bacteria from environment (concentration varies by biome, temperature, moisture) colonize wound. Untreated: infection spreads systemically → sepsis → death
- Treatment: heat (cauterize), antiseptic plants (some reduce bacterial growth rate), clean bandaging (reduces exposure)

### Atmosphere
**Outdoors:** O₂ is effectively unlimited at sea level. Concentration decreases with altitude using the real barometric formula. Above 4000m, O₂ is noticeably reduced; above 8000m, survival without adaptation is limited.

**Enclosed spaces (rooms/caves):** Cavity detection via flood-fill algorithm identifies enclosed volumes. Each enclosed cavity has a total O₂ budget based on volume. Fire consumes O₂ from cavity budget. When O₂ drops below 16%, player loses consciousness. Below 12%, death.

Scope boundary: outdoors = altitude-based atmospheric model (per-region). Enclosed = cavity O₂ budget (per-room, not per-cell, for performance).

### What stays precomputed
Geology, terrain shape, ore placement, biome layout, world civilization history — all generated from seed before players join. Real-time simulation handles only active processes.

## Section 3: Player & Survival

### The survival loop
The player is a human. Every mechanic connects to the simulation.

| Need | Real basis | Consequence if ignored |
|------|-----------|----------------------|
| Hunger | Glucose/ATP depletion rate (body size and activity-scaled) | Slowing, weakness, death |
| Thirst | Cellular dehydration (faster than hunger) | Critical in heat and exertion |
| Temperature | Body heat vs ambient cell temperature | Hypothermia in cold, hyperthermia in heat |
| Sleep | Cognitive and physical restoration | Slower reactions, poor aim, eventually hallucinations |
| Wounds | Tissue damage + bacterial infection | Infection spreads → sepsis → death |
| Oxygen | Atmospheric O₂ concentration from sim grid | Relevant at altitude and in enclosed spaces |

### Sleep (multiplayer rules)
- Player can sleep only if they have a placed bedroll or are inside an owned shelter
- Sleep does NOT advance server time — other players continue in real time
- While sleeping, player is fully vulnerable — can be killed, looted
- Sleep restores stamina and cognitive stat at a rate based on shelter quality
- Minimum meaningful rest: 30 seconds real-time
- Sleeping in the open (no bedroll) gives no restoration and exposes player to environmental temperature

### Character progression
No XP bars. No level-up menus. Skills improve by doing them:
- More hunting → better aim and tracking ability
- More smithing → faster work, less material waste, higher output quality
- More medicine → faster wound treatment, higher success rate against infection
- More construction → faster building, access to more complex structural configurations

**Skill × chemistry interaction:**
Skill affects speed and precision of player actions — not the chemistry itself. The chemistry engine produces the same results regardless of skill. Bronze always forms at the same eutectic point. What skill changes:
- Action speed (a skilled smith works faster)
- Material waste (a novice wastes more ore per smelting run)
- Output quality (a skilled smith's blade holds an edge longer due to better carbon distribution — modeled as a quality multiplier on the item)

### Combat
Real physics. Weapons have weight, reach, momentum:
- Heavy axe: slow swing, high damage, knocks targets
- Knife: fast, short range, good for stealth
- Spear: long reach, requires space, thrown ballistics follow real arc
- Arrows: real ballistic trajectory including wind resistance
- Armor: material properties matter — bronze deflects differently than iron, leather absorbs blunt force
- Single well-placed hit can kill. No health bar that absorbs 30 sword swings.
- Wounds from combat are real wounds that require the biology/pathogen system to treat.

### Death (reconciled with PvP outlaw system)
- On death: drop all inventory items at death location. Body persists in world and can be looted for a short window (5 minutes).
- Respawn: at last placed bedroll if it still exists, otherwise at world spawn zone.
- Death does NOT clear criminal record from NPC memory. NPCs recognize you by appearance and record.
- Criminal record reset mechanism: completing NPC-assigned tasks (bounties, service) can reduce standing over time.

### Roles (emergent, not chosen)
No class selection. Role emerges from what you actually do:
- Gather herbs + study the chemistry of disease → healer
- Master iron smelting → the person other players need
- Recruit followers + build walls → warlord
- Trade between settlements → merchant

The world doesn't assign these. Other players and NPCs recognize them through actions and reputation.

### Full vertical slice requirement
**Ordering and pass/fail criteria:**

| Slice | Components | Pass criteria |
|-------|-----------|---------------|
| 1 | Gather loose rock/wood from surface | Playtester: walks to resource, presses F, item appears in inventory slot, can be inspected. Zero errors. |
| 2 | Craft stone tool, equip it, use it | Playtester: has rock + flint in inventory, performs craft action, tool appears, equips to hand slot, can chop a tree with it (tree loses health/falls). Zero errors. |
| 3 | Light fire (chemistry-driven) | Playtester: gathers tinder + wood, strikes flint, fire ignites in sim grid, produces heat cells, nearby temperature rises. Zero errors. |
| 4 | Cook food, eat it, hunger restores | Playtester: has raw food, places near fire, food reaches cooking temperature, becomes cooked, eat action restores hunger bar. Zero errors. |
| 5 | Wound, infection, treatment | Playtester: takes damage, wound appears, bacteria count rises over time without treatment, applying herb reduces bacteria, wound heals. Zero errors. |
| 6 | Build shelter, sleep in it, stamina restores | Playtester: gathers materials, places walls + roof, places bedroll inside, sleeps, stamina increases. Zero errors. |
| 7 | Smelt copper, craft copper tool | Playtester: builds furnace, places copper ore + charcoal, lights fire, copper appears, crafts copper knife, equips it. Zero errors. |

**After every slice:** automated playtester agent plays the slice end-to-end. Fix any failures. Re-run until clean. Only then start next slice.

## Section 4: Multiplayer & Society

### Scale target
- **50–200 concurrent players** per world instance in v1
- Server is authoritative for: simulation grid state, NPC state, inventory persistence
- Client is authoritative for: player input (movement via KCC, action triggers)
- Rapier physics runs server-side too (authoritative collision), client does local prediction with server correction

### Shared simulation
- Everything happens in the same world. Burning a forest removes it for everyone.
- Damming a river changes downstream water flow for all players.
- Resources are finite and deplete. A rich iron vein mined for months has less ore.
- Shared consequences are central to the social structure — they force cooperation and create conflict.

### Social structure (emergent)
No enforced factions or teams:
- Players who settle together and build walls form a de facto settlement
- Settlements controlling resources attract followers or enemies
- Trade routes form because both sides benefit from real resource specialization (region with copper but no tin needs someone who has tin)
- Conflict happens when territories overlap or someone wants what you have

### NPC AI
**Architecture: Utility AI with needs-based drives**

Each NPC has a set of needs with current values (hunger, safety, shelter, social belonging). At each tick, a utility function scores all available actions against all needs. The highest-scoring action is taken.

NPC needs:
| Need | What drives it | What satisfies it |
|------|--------------|------------------|
| Hunger | Time + activity | Eating food |
| Safety | Nearby threats (players with criminal record, predators, night) | Shelter, distance from threats, allies |
| Shelter | Cold, rain, night | Being inside structure |
| Social | Time alone | Being near settlement members |
| Economic | Low resources | Trade, gathering, crafting |

**NPC memory:**
- NPCs remember player interactions (trust/threat model per player)
- Trust increases via: trade, helping NPC tasks, defending settlement
- Threat increases via: attacking NPCs, stealing, high criminal record
- Memory persists in Neon DB — NPCs remember players across server restarts

**NPC civilization:**
- Settlements have a shared "civilization level" that advances as members research and craft
- NPCs craft tools, gather, build, and advance organically using the same physics/chemistry rules as players
- They do not advance magically — they are subject to the same simulation constraints

### PvP and outlaw system (reconciled with death model)
Full PvP:
- Players can attack and kill other players at any time
- Killing another player adds a "Murder" mark to your criminal record (visible to all NPCs)
- Criminal record thresholds:
  - 1 murder: NPCs are cautious, prices increase in shops
  - 3 murders: NPC settlements close gates, refuse trade
  - 5+ murders: active bounty posted — other players see reward amount, NPC guards attack on sight
- Bounties accumulate until the outlaw dies or completes redemption tasks
- Death does NOT clear the criminal record (NPCs recognize you on respawn)
- Redemption: completing NPC-assigned service tasks (escort, resource delivery, defending settlement) reduces criminal standing over time — the RDR2 model

## Section 5: Building System

### Overview
Players construct physical structures from gathered materials. Buildings are composed of placed blocks/pieces (wall segments, floors, roofs, doors). Structures interact with the simulation engine — they are not cosmetic.

### Material properties in building
| Material | Structural behavior | Thermal behavior | Fire behavior |
|----------|--------------------|-----------------|----|
| Wood | Moderate load capacity | Insulates (low conductivity) | Burns (O₂ + heat → combustion) |
| Stone | High load capacity | Poor insulator (high conductivity) | Does not burn |
| Iron/Bronze | Very high load | High conductivity | Does not burn, melts at extreme heat |
| Clay/Brick | Moderate | Moderate insulation | Does not burn |

### Physics constraints
- Gravity applies to all placed structures. Unsupported blocks fall.
- Load transfer: heavy materials need structural support below them (stone on wood = eventual collapse)
- No floating structures

### Simulation integration
- Fire spreads to wood structures when adjacent cell temperature exceeds wood ignition point (~300°C)
- Temperature conducts through walls based on material conductivity — a stone wall slows heat loss, wood insulates better
- Enclosed spaces trigger the cavity O₂ system (see Section 2)
- Rain and moisture affect wood degradation over time (slow)

### Ownership
- The player who places a structure block owns it
- Owned blocks can only be removed by owner or by combat damage
- NPC settlement buildings are NPC-owned — players cannot remove them without hostile action

## Section 7: Companion Website

### What it is
A separate deployed web application — not embedded in the game client — that acts as a real-science field guide for players. Players who don't understand why something works a certain way in the game can ask and get a real explanation.

### Input model
Free-text questions. Player types: "Why did my fire go out inside the cave?" The assistant explains the enclosed space O₂ system in plain language, tied to real atmospheric science.

### AI backing
- Backed by Claude API
- System prompt grounds the assistant in the game's real-world scientific context
- Guardrails: explains real physics/chemistry/biology only. Does not hallucinate game mechanics. If a mechanic isn't implemented yet, says so rather than inventing an answer.
- Content scope matches implemented game features — grows as game grows

### Data access (v1)
- Does NOT have access to the player's live game state in v1
- Players describe their situation in text; assistant responds based on general science + game rules
- Future: can connect to game API to read player's current biome, inventory, vitals for contextual answers

### Persistence
- Conversation history stored per session (not per player in v1 — no login required on companion site)
- Each generation stored with a unique ID for debugging and quality review

### Deployment
- Separate Vercel deployment from the main game
- Linked from the game HUD (help icon opens companion site in new tab)

## Implementation Rules

1. **Full vertical slices only** — no feature is "done" until it is end-to-end functional per the pass/fail criteria in Section 3
2. **Playtester loop after each slice** — automated agent plays the slice, reports broken steps, developer fixes, agent re-tests until clean pass. No exceptions.
3. **No dead ends** — if a UI element exists, it must do something real. Do not build panels or buttons for unimplemented backend systems.
4. **No UI-only features** — do not build inventory/crafting/equip UI without the backend systems connected
5. **The simulation is the source of truth** — never hardcode outcomes. Let the physics/chemistry/biology engine produce results.
6. **Test-driven** — write the acceptance test (playtester scenario) before implementing each slice
